## 1. Реализация Redis-кэша

Реализация: `backend/app/application/cache_service.py`, эндпоинты — `backend/app/api/cache_demo_routes.py`, ключи и TTL — `backend/app/infrastructure/cache_keys.py`.

### Что кэшируется

| Данные | Назначение ключа |
|--------|------------------|
| **Агрегат каталога** | Агрегация по `order_items`: название товара, число строк, сумма `subtotal` по каждому `product_name`. Используется для демонстрации «каталога» без тяжёлого JOIN по всем заказам при каждом запросе. |
| **Карточка заказа** | Снимок заказа: поля заказа из таблицы `orders`, позиции из `order_items`, сериализованный JSON. |

### Ключи в Redis

| Ключ | Формат |
|------|--------|
| Каталог | `catalog:v1` |
| Карточка заказа | `order_card:v1:{order_id}` — один ключ на каждый `order_id` (UUID в строке). |

### TTL

- Константа `DEFAULT_CACHE_TTL_SECONDS = 300` (5 минут).
- При записи в Redis: `SET ... EX 300` для каталога и карточки.

### Логика cache hit / miss

- **Hit:** для запроса с `use_cache=true` выполняется `GET` по ключу; если значение есть — возвращается десериализованный JSON **без** обращения к PostgreSQL.
- **Miss:** ключа нет или `use_cache=false` — выполняется SQL (`_fetch_catalog_rows` или `_fetch_order_card_from_db`), результат при `use_cache=true` кладётся в Redis с TTL; при `use_cache=false` в Redis запись **не** выполняется (удобно для бенчмарков «холодная» БД).

Параметр **`use_cache`** передаётся query-параметром в `/api/cache-demo/catalog` и `/api/cache-demo/orders/{order_id}/card`.

## 2. Демонстрация неконсистентности (намеренно сломанный сценарий)

**Эндпоинт:** `POST /api/cache-demo/orders/{order_id}/mutate-without-invalidation`  
Тело: JSON `{"new_total_amount": <число>}`. Обновляется только строка заказа в БД (`UPDATE orders SET total_amount = ...`). **Инвалидизация Redis не вызывается.**

### Шаги сценария

1. **Прогрев кэша** — несколько раз `GET /api/cache-demo/orders/{order_id}/card?use_cache=true` (или один раз после создания заказа), чтобы в Redis появился ключ `order_card:v1:{order_id}` с актуальным `total_amount`.
2. **Изменение в БД без инвалидации** — `POST .../mutate-without-invalidation` с новым `new_total_amount`. В Redis старый JSON карточки остаётся.
3. **Повторный запрос «как у пользователя»** — снова `GET .../card?use_cache=true`: ответ берётся из Redis → клиент видит **старое** значение суммы и состава из кэша.
4. **Сравнение с истиной** — `GET .../card?use_cache=false` или прямой запрос к БД показывает уже **новое** `total_amount` после `UPDATE`.

### Чем stale отличается от БД

В объекте `order_card` из кэша поле **`total_amount`** (и при необходимости производные поля), соответствующее состоянию **до** шага mutate, не совпадает с фактическим значением в таблице `orders` после обновления. Автоматических признаков ошибки нет — ответ HTTP 200; неконсистентность проявляется только сравнением с данными без кэша или с БД.

Автоматическая проверка сценария: тест `backend/app/tests/test_cache_stale_consistency.py`.

## 3. Починка через событийную инвалидацию

### Где генерируется событие

После успешного `UPDATE` заказа в эндпоинте **`POST /api/cache-demo/orders/{order_id}/mutate-with-event-invalidation`** вызывается:

`CacheInvalidationEventBus.publish_order_updated(OrderUpdatedEvent(order_id=...))`.

Реализация варианта **C** из README: синхронная публикация сразу после изменения строки в БД (отдельная таблица outbox в БД не используется; файл `backend/migrations/003_cache_invalidation_events.sql` содержит только опциональный черновик схемы для расширения до outbox/worker).

### Где обрабатывается событие

Файл **`backend/app/application/cache_events.py`**: класс `CacheInvalidationEventBus`, метод `publish_order_updated`. Обработка — удаление ключей в Redis через асинхронный клиент `get_redis()`.

### Какие ключи инвалидируются и зачем

| Ключ | Причина |
|------|---------|
| `order_card:v1:{order_id}` | Карточка конкретного заказа изменилась; старый JSON недопустим. |
| `catalog:v1` | Каталог агрегирует данные по `order_items`; при изменении заказа могли измениться выручка и счётчики по товарам, поэтому глобальный ключ каталога сбрасывается (консервативная инвалидация агрегата). |

После инвалидации следующий запрос с `use_cache=true` снова строит ответ из БД и при необходимости заполняет кэш.

Проверка: тест `backend/app/tests/test_cache_event_invalidation.py`.

## 4. Rate limiting endpoint оплаты через Redis

**Файл:** `backend/app/middleware/rate_limit_middleware.py`.

### Policy (лимит)

- **Не более 5 запросов** за **скользящее окно 10 секунд** на один «subject» (`limit_per_window=5`, `window_seconds=10`).
- Учитываются только **POST**-запросы к:
  - `/api/orders/{uuid}/pay`;
  - `/api/payments/retry-demo`.

### Ключ в Redis

- Шаблон: `rate_limit:pay:{subject}` (`payment_rate_limit_key` в `cache_keys.py`).
- **subject:** приоритетно первый IP из заголовка `X-Forwarded-For`, иначе `request.client.host`, иначе строка `unknown`.

### Поведение при превышении

- Возвращается **HTTP 429 Too Many Requests**, тело JSON: `{"detail": "Too Many Requests"}`.

### Заголовки ответа

При срабатывании и при успешных запросах в пределах лимита добавляются (или дополняют ответ):

- `X-RateLimit-Limit` — лимит (5);
- `X-RateLimit-Remaining` — оставшееся число запросов в окне;
- `X-RateLimit-Reset` — приблизительный Unix timestamp сброса окна.

Проверка: `backend/app/tests/test_payment_rate_limit_redis.py`.

## 5. Бенчмарки RPS до/после кэша

### Условия замеров

- Инструмент: **wrk** на хосте (WSL), параметры: **`-t4 -c100 -d30s`**, базовый URL **`http://localhost:8082`** (при другом порте на хосте подставить свой).
- Перед прогоном карточки заказа: подготовка данных, например  
  `docker compose exec -T db psql -U postgres -d marketplace -f /sql/04_wrk_benchmark_order.sql`.
- Скрипты: с кэшем — `loadtest/wrk/catalog.lua`, `loadtest/wrk/order_card.lua` (`use_cache=true`); без кэша — `catalog_no_cache.lua`, `order_card_no_cache.lua` (`use_cache=false`).

### Результаты wrk (один из прогонов)

**С кэшем (`use_cache=true`), прогретый Redis после нагрузки:**

| Endpoint (логика в .lua) | Requests/sec | Avg latency | Socket timeout |
|--------------------------|-------------|-------------|----------------|
| `GET /api/cache-demo/catalog?use_cache=true` | **1132.16** | **96.98 ms** | 100 |
| `GET /api/cache-demo/orders/{id}/card?use_cache=true` | **1105.19** | **99.20 ms** | 100 |

**Без кэша (`use_cache=false`):**

| Endpoint | Requests/sec | Avg latency | Socket timeout |
|----------|-------------|-------------|----------------|
| `GET .../catalog?use_cache=false` | **558.89** | **222.32 ms** | 100 |
| `GET .../card?use_cache=false` | **494.68** | **265.79 ms** | 105 |

Ответы в замере — успешные (строки `Non-2xx or 3xx` в выводе wrk отсутствовали). Отдельные `timeout` на фоне десятков тысяч запросов можно трактовать как редкие при 100 одновременных соединениях.

### Сравнение

| Сценарий | RPS с кэшем | RPS без кэша | Относительный выигрыш по RPS (каталог / карточка) |
|----------|-------------|--------------|--------------------------------------------------|
| Каталог | 1132 | 559 | ~**+102%** (~в 2 раза) |
| Карточка | 1105 | 495 | ~**+123%** (~в 2.2 раза) |

Средняя задержка при отключённом кэше выросла примерно в **2–2.7 раза** относительно сценария с Redis.

**Альтернатива:** Locust (`loadtest/locustfile.py`) — в одном прогоне раздельная статистика по именам задач `[cache on]` / `[cache off]` для каталога и карточки.

## 6. Выводы

1. **Когда кэш даёт выигрыш.** Для часто запрашиваемых чтений (агрегат каталога, карточка заказа) Redis снижает нагрузку на PostgreSQL и даёт заметный рост RPS и снижение средней задержки при типичных параметрах wrk; выигрыш хорошо виден при сравнении `use_cache=true` и `use_cache=false` при одинаковой нагрузке.

2. **Когда выигрыш может быть меньше.** Если данные редко читаются, TTL очень короткий, или узкое место — не БД, эффект от кэша будет скромнее; при полном промахе кэша добавляется один round-trip к Redis без выигрыша по данным.

3. **Инвалидация сложнее кэширования.** Запись в кэш локальна и предсказуема; при изменении доменных данных нужно явно определить, какие ключи удалить или обновить. Пропуск инвалидации даёт устаревшие ответы при HTTP 200; выбран вариант синхронного сброса `order_card` и `catalog` по событию обновления заказа.

4. **Rate limiting и бизнес-логика.** Даже при проверках в сервисе ограничение частоты на уровне HTTP и Redis защищает от всплесков, брутфорса и случайных повторных кликов по оплате, возвращая 429 и понятные заголовки лимита.

5. **Тестирование.** Проверочные тесты ЛР5 (`test_cache_stale_consistency`, `test_cache_event_invalidation`, `test_payment_rate_limit_redis`) закрепляют сценарии stale cache, событийной инвалидации и лимита оплаты; при разработке важно согласовывать event loop pytest с async SQLAlchemy/Redis (как настроено в проекте).
